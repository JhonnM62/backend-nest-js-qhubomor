# Backend Deployment Guide

This document outlines the CI/CD pipeline, Docker configuration, and server setup for the Q Hubo Mor POS backend.

## Infrastructure
- **Server**: Contabo VPS (Ubuntu/Debian)
- **Containerization**: Docker & Docker Compose
- **Reverse Proxy**: Nginx (routing `/api/v1` to the Docker container)
- **CI/CD**: GitHub Actions

## GitHub Actions Workflow (`deploy.yml`)
Every push to the `main` branch triggers an automated deployment:
1. **Checkout**: Pulls the latest code.
2. **SSH Connection**: Connects to the Contabo VPS using secrets (`HOST`, `USERNAME`, `PASSWORD`, `PORT`).
3. **Repository Pull**: Pulls the latest changes directly on the server.
4. **Environment Setup**: Injects the `.env` file from GitHub Secrets.
5. **Docker Build**: Rebuilds the `qhubomor-backend` image.
6. **Directory Provisioning**: Creates persistent directories on the host machine (`/root/backend-qhubomor/public/uploads/...`).
7. **Container Restart**: Stops the old container and runs the new one, mapping ports and volumes.

## Docker Configuration (`Dockerfile`)
The backend uses a multi-stage Docker build to keep the final image lightweight.
- **Base Image**: `node:20-slim`. *CRITICAL*: Do not use `alpine`. Prisma's Query Engine requires OpenSSL 1.1/3.0 and glibc, which conflicts with Alpine's musl libc.
- **Dependencies**: `RUN apt-get update && apt-get install -y openssl` is mandatory.
- **Build Step**: Runs `npx prisma generate` and `npm run build`.

## Persistent Storage (Volumes)
To prevent data loss (images, receipts) when the Docker container is destroyed and recreated during deployment:
- **Host Path**: `/root/backend-qhubomor/public/uploads`
- **Container Path**: `/app/public/uploads`
- **Docker Run Command**: `-v /root/backend-qhubomor/public/uploads:/app/public/uploads`

## Nginx Proxy Rules
Nginx acts as the gateway on the VPS. It is configured to intercept `/api` requests and forward them to the Docker container running on port `8061`.
Because of this, the NestJS `ServeStaticModule` MUST serve static files under the `/api/v1` prefix to ensure URLs generated by the backend are accessible through Nginx.
